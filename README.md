# zt-ui

`zt-ui` is an open-source Zig immediate-mode dashboard reference. The project is intentionally opinionated: browser-hosted WASM first, command-buffered rendering, plain-data app state, and an operations-console reference app instead of a generic widget toolkit.

## Architecture

```text
JS platform
  ↓
Zig app runtime (WASM)
  ↓
Render backend
  ↓
Frame arena + command list
  ↓
UI layer
  ↓
App code
```

That architecture maps directly to the source tree:

| Layer          | Role                                                                                                               |
| -------------- | ------------------------------------------------------------------------------------------------------------------ |
| `src/platform` | Browser boundary, input snapshots, frame timing, and WASM exports.                                                 |
| `src/gfx`      | Shared geometry types, command records, transient frame buffers, and renderer-facing payload storage.              |
| `src/ui`       | Stable widget ids, stack layout, clipping helpers, theme, and immediate-mode widgets.                              |
| `src/app`      | The reference app: state, panel geometry, charts, and dashboard composition.                                       |
| `src/debug`    | Overlay and frame metrics for developer-facing runtime inspection.                                                 |
| `web/`         | The browser host, runtime boot sequence, and a Canvas2D renderer living behind the future-facing `webgpu.js` seam. |

## What This Repo Optimizes For

- Immediate-mode dashboard first, not a general-purpose toolkit.
- A serious reference surface for workflow software: benchmark metrics, pipeline throughput, activity logs, and a debug overlay.
- Test-first seams for layout, ids, renderer payloads, input handling, and state updates.
- Explicit transient memory via per-frame command, text, and point buffers.
- A browser host that can evolve from Canvas2D to WebGPU without changing the Zig-side app model.

## Current Status

Already real:

- `zig build` compiles the WASM runtime and syncs `web/app.wasm`.
- `zig build test` exercises the shared Zig module tree locally.
- The runtime exports a browser-oriented surface for resize, frame stepping, input events, and frame-buffer access.
- The browser host forwards CSS-pixel input and draws the returned command list onto a canvas.
- The reference app renders an immediate-mode workflow console with metrics, charting, logs, and a debug overlay.

Still intentionally conservative:

- `webgpu.js` is currently Canvas2D-backed while the command ABI and text path settle.
- The renderer is command-buffer oriented, but not yet GPU-batched.
- Text uses the browser’s font stack instead of a Zig-side glyph atlas.
- The chart system is intentionally single-purpose and tuned for the reference app rather than generalized.

## Project Standards

- The public tone should stay sober and maintainable. Favor clarity over hype.
- The runtime contract between Zig and the browser host is part of the API surface.
- Test coverage should grow before abstraction does.
- Reusable toolkit pieces should be extracted only after the dashboard proves them.

## Build And Run

Requires Zig `0.15.2` or newer.

```sh
zig build
zig build wasm
zig build test
zig build serve
```

Then open `http://127.0.0.1:8080`.

`web/app.wasm` is a local build artifact generated by `zig build`; it is intentionally ignored in version control.
`zig build serve` rebuilds `web/app.wasm` first, then serves the `web/` directory with the repo's Zig dev server.
Use `zig build serve -- 8091` if you want a different local port.

## Contributing

The goal is to keep the repo useful as a public reference, not let it drift into framework abstraction for its own sake.

- Prefer plain data and explicit flow over hidden magic.
- Add tests when you harden a seam or add a rule.
- Keep the app layer concrete until there is repeated pressure to extract a reusable primitive.
- Preserve the command-buffer boundary between UI code and rendering.
